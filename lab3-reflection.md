Lab 3 – AWS Cloud Security Reflection

In this lab, I set up a basic AWS environment and focused on securing it using IAM, S3, EC2, and monitoring tools. The overall purpose was to see how cloud security works in a real-world setting and to better understand how breaches like Capital One can happen. One thing that became clear to me is that the cloud itself is not automatically insecure most problems happen because of misconfigurations or weak monitoring.

For IAM, I created a separate IAM user and group instead of continuing to use the root account. I gave the user only the permissions needed for the lab. This helped me really understand the idea of least privilege. It sounds simple in theory, but actually setting permissions and seeing what happens when access is denied made it more real. In the Capital One breach, the attacker was able to access sensitive data because an IAM role had overly broad permissions. If stricter access controls had been in place, the breach likely would not have been as severe. Creating a limited IAM user showed me why using root casually is risky and why permissions should always be kept as narrow as possible.

For S3, I created a bucket and made sure that “Block Public Access” was enabled. I double-checked the permissions tab to confirm that nothing was exposed. This part stood out to me because it would be very easy to overlook one small setting and accidentally make data public. Even though the Capital One breach was not caused by a public S3 bucket, it still involved weak access controls once the attacker gained access. This reinforced how important it is to review bucket policies and access settings carefully.

I also launched a Free Tier EC2 instance and configured the security group so that SSH was only allowed from my personal IP address instead of 0.0.0.0/0. That small change makes a big difference because it limits who can even attempt to connect. In the Capital One case, the attacker used a vulnerability (SSRF) to access the EC2 metadata service and retrieve credentials. While we did not configure a WAF in this lab, it made me realize how network rules and instance configurations directly affect security.

When I explored monitoring tools, I checked the Billing Dashboard and attempted to access CloudTrail. My IAM user did not have full permissions, which caused some “Access Denied” messages. At first it was frustrating, but it actually demonstrated least privilege working correctly. CloudTrail logs account activity, which is critical for detecting suspicious behavior. One of the major issues in the Capital One breach was that the attacker remained undetected for a period of time. Monitoring tools like CloudTrail, AWS Config, and billing alerts help reduce that risk by increasing visibility.

Some parts of the lab were straightforward, like launching resources and capturing screenshots. The most confusing part was managing permissions. It was surprisingly easy to either give too much access or lock myself out. That experience showed me how small mistakes in configuration can create serious security gaps.

This lab helped me connect the technical steps to real-world consequences. IAM, S3, EC2, and monitoring are all connected, and security depends on how carefully each piece is configured and reviewed. The biggest takeaway for me is that cloud security is not just about having the right tools — it’s about using them correctly and consistently.
