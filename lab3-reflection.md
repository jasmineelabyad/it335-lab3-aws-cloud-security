Lab 3 – AWS Cloud Security Reflection
In this lab I set up an AWS environment and secured IAM S3 EC2 and monitoring features. The goal was to understand how configuration decisions in AWS connect to the Capital One breach. That incident showed that the cloud itself was not the problem. The issue came from misconfiguration broad permissions and weak detection. Small setup decisions led to serious consequences.

I first checked the root account and confirmed that multi factor authentication was enabled. After that I created a separate IAM user and stopped using root for normal tasks. I did not assign AdministratorAccess. I only granted the permissions needed to complete the lab. This follows least privilege. Least privilege means access should be limited to what is necessary. In the Capital One breach an EC2 instance role had permissions that were too broad. When the attacker exploited SSRF and accessed the metadata service those permissions allowed access to S3 data. If the permissions had been more limited the attacker would not have been able to retrieve as much information. Identity configuration directly affects how much damage can happen after credentials are exposed.

I then created an S3 bucket and made sure that Block all public access was turned on. I reviewed the permissions tab to confirm that the bucket was not publicly available. S3 misconfiguration is a common cause of cloud data exposure. Even if access is not intentional incorrect settings can expose data. In the Capital One case the attacker used stolen credentials rather than a public bucket. However storage controls still mattered. Once inside the account weak controls allowed data retrieval. Blocking public access and reviewing bucket settings reduces the chance of accidental exposure.

I also launched a Free Tier EC2 instance and configured a security group. SSH access was restricted to my own IP address. I did not allow 0.0.0.0/0. Open inbound rules increase the attack surface and make it easier for unauthorized users to attempt connections. I reviewed metadata settings and noted the importance of IMDSv2. In the Capital One breach the attacker accessed IMDSv1 through an SSRF vulnerability and retrieved temporary credentials. That event shows how network exposure and instance settings connect to identity permissions. If network rules are restricted and metadata protections are stronger the impact of an SSRF style attack can be reduced.

I reviewed CloudTrail and the Billing Dashboard to understand how AWS tracks activity and usage. CloudTrail logs API calls and account activity. Billing shows spending patterns and unusual changes. In the Capital One breach the attacker operated for a period of time before detection. When organizations rely only on periodic audits there is a delay between activity and review. That delay creates a window of opportunity. During that time attackers can move through systems without being noticed. Continuous monitoring shortens that window because logs are generated consistently and can be reviewed quickly. Faster detection reduces the overall impact of an incident.

The case also shows that having tools is not enough. Security depends on governance and oversight. Permissions must be reviewed and monitoring must be active. Configurations must be checked regularly. Without that even basic controls can fail. The Capital One breach demonstrates how misconfiguration combined with weak oversight can lead to major data exposure.

Overall this lab showed how identity controls storage permissions network restrictions and monitoring work together. Each configuration choice affects risk. Applying least privilege blocking public access limiting inbound traffic and using continuous logging reduces the likelihood and severity of cloud security incidents.
